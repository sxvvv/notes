## 1. GPU 内存层次结构概述

与 CPU 类似，GPU 也采用多级内存层次结构，以确保流多处理器（SM）能够快速获取数据。
下图以 NVIDIA Tesla V100 为例展示了完整的 GPU 内存层次：

<img width="1636" height="1380" alt="image" src="https://github.com/user-attachments/assets/acb3c640-a0b1-4f4a-8b52-5a8739ce3214" />


- 最靠近 SM 的片上内存：寄存器文件、L1 缓存、共享内存、常量缓存。

- 片外内存：L2 缓存、全局内存、局部内存、纹理内存、常量内存（物理上位于显存中）。

- 数据传输路径：数据通过 PCIe 或 NVLink 进入设备，首先经过 L2 缓存，然后进入全局内存；在 SM 内部，数据通过各级缓存最终到达寄存器供 CUDA 核心使用。

GPU 的缓存行大小为 128 字节（32 个线程 × 4 字节），正好匹配一个 warp 的线程数，使得一次加载可以服务整个 warp。

## 2. 详细内存类型

### 2.1 片上内存（每个 SM 私有）

| 内存类型 | 位置与作用 |
|----------|------------|
| **寄存器文件** | 每个 SM 拥有大量寄存器（如 V100 有 65536 个 4 字节寄存器）。寄存器文件组织为 32 个 bank，对应 warp 的 32 个线程。每个线程独享寄存器，访问延迟极低。 |
| **L1 缓存** | 用于缓存最近从全局内存读取或写入的数据。当寄存器不足时，发生“寄存器溢出”，多余数据会存入 L1 缓存（若 L1 也满，则继续溢出到局部内存）。L1 与共享内存物理上位于同一块存储区域，可通过应用程序划分比例。 |
| **共享内存** | 同一线程块内所有线程可访问的片上内存，用于线程间通信和数据共享。需显式声明和使用。 |
| **常量缓存** | 用于缓存全局内存中声明为 `__constant__` 的只读变量。最佳用途是广播单个常量值给 warp 中所有线程。 |

### 2.2 片外内存（所有 SM 共享）

| 内存类型 | 位置与作用 |
|----------|------------|
| **L2 缓存** | 片上统一缓存，所有 SM 共享，用于加速数据在 SM 与全局内存之间的传输。同时位于 PCIe/NVLink 的数据路径上。 |
| **全局内存** | GPU 的主内存（显存），容量最大但延迟最高。V100 使用 HBM2，RTX 5000 使用 GDDR6。 |
| **局部内存** | 主内存中为每个 SM 映射的专用区域。当寄存器溢出超出 L1 容量时，数据会先到 L2，再到局部内存。访问局部内存代价很高。 |
| **纹理内存** | 只读内存区域，通过纹理缓存（L1）访问，适合具有空间局部性的访问模式（如图形渲染）。 |
| **常量内存** | 只读内存区域，通过常量缓存访问，用于存储常量数据。 |

> 注意：纹理内存和常量内存虽然物理上位于全局内存，但由于有专用缓存，访问效率可能更高。

## 3. GPU 与 CPU 内存对比

以 NVIDIA Tesla V100（Volta 架构）与 Intel Xeon Cascade Lake/Skylake 处理器为例，对比每 SM 与每核心的内存配置。

### 3.1 容量对比

| 内存类型 | NVIDIA Tesla V100（每 SM） | Intel Xeon（每核心） |
|----------|-----------------------------|----------------------|
| **寄存器文件** | **256 kB** | 10.5 kB |
| **L1 缓存** | **128 kB**（最大） | 32 kB |
| **常量缓存** | 64 kB | - N/A - |
| **L2 缓存** | 0.075 MB（共享 L2 的分片） | **1 MB**（私有） |
| **L3 缓存** | - N/A - | **1.375 MB**（共享分片） |
| **全局内存/DRAM** | 0.4 GB（每 SM 分片） | **>3.4 GB**（每核心分片） |
 
- GPU 在最近内存层（寄存器、L1）的容量远大于 CPU，但越往外层（L2、全局）CPU 优势越明显。
  
- GPU 通过大容量寄存器文件支持大量线程，而 CPU 依靠大缓存降低延迟。

### 3.2 延迟与带宽对比

| 内存类型 | NVIDIA Tesla V100（每 SM） | Intel Xeon（每核心） |
|----------|-----------------------------|----------------------|
| **L1 缓存** | 延迟 28 周期，带宽 128 B/周期 | 延迟 **4 周期**，带宽 **192 B/周期** |
| **私有 L2** | - N/A - | 延迟 14 周期，带宽 64 B/周期 |
| **共享 L2/L3** | 延迟 193 周期，带宽 17.6 B/周期 | 延迟 **50-70 周期**，带宽 14.3 B/周期 |
| **全局内存/DRAM** | 延迟 220-350 周期，带宽 **7.4 B/周期** | 延迟 190-220 周期，带宽 1.9-2.5 B/周期 |

- CPU 在各级缓存上延迟更低，带宽更高。

- GPU 的全局内存带宽远高于 CPU，但延迟也更高。因此 GPU 需要通过大量线程来隐藏延迟（延迟隐藏技术）。

## 4. 延迟隐藏与数据传输优化

### 4.1 延迟隐藏（Latency Hiding）

- **问题**：全局内存访问延迟高达数百周期，若单个线程等待数据，会闲置计算资源。

- **解决方案**：启动远多于物理核心的线程。当一个 warp 因访存而停滞时，SM 的调度器立即切换到另一个就绪的 warp 执行，从而隐藏延迟。

- **关键**：足够的线程数量是发挥 GPU 性能的前提。

### 4.2 主机与设备数据传输

- GPU 通过 PCIe（或 NVLink）与主机连接，传输延迟高。

- **优化策略**：

  - 尽量批量传输大块数据，减少传输次数。

  - 使用 CUDA 流（streams）将数据传输与内核执行重叠，实现计算与通信的并行。

  - 尽可能将数据保留在设备端，避免频繁往返。

## 5. 更细粒度的内存切片（CUDA 核心 vs CPU 矢量通道）

为了更精确地比较，我们考虑每个 CUDA 核心与每个 CPU 矢量通道可用的内存资源。

### 5.1 寄存器文件

- **V100**：每 SM 65536 个寄存器，每 SM 每周期处理 2 个 warp（即 64 个线程），因此每 CUDA 核心可用寄存器数 ≈ 65536 / 64 = 1024 个浮点数。

- **Xeon**：每核心有 168 个 64B 矢量寄存器（总 10.5 kB），每核心每周期可处理 2 个 16-float 矢量（即 32 个浮点通道），因此每矢量通道可用寄存器数 ≈ (168×16) / 32 = 84 个浮点数。

  - **结论**：CUDA 核心的寄存器资源比 CPU 矢量通道多一个数量级。

### 5.2 缓存

- **V100**：每 SM 拥有 L1 128 kB + 常量 64 kB + L2 分片 0.075 MB = 约 0.26 MB，除以每 SM 的 64 个 CUDA 核心，得 **每核心 0.0041 MB**。

- **Xeon**：每核心 L1 32 kB + L2 1 MB + L3 分片 1.375 MB = 2.4 MB，每核心有 32 个矢量通道，得 **每通道 0.075 MB**。

  - **结论**：CPU 每矢量通道的缓存资源比 CUDA 核心多两个数量级。

### 5.3 缓存行与扇区

- **GPU**：缓存行 128 字节，分为 4 个 32 字节扇区。缓存未命中时只需加载需要的扇区，而非整行，提高了传输效率。

- **CPU**：缓存行 64 字节，必须整行加载。

---

## 总结

- GPU 内存层次以大量寄存器和高带宽全局内存为特点，通过延迟隐藏技术容忍高延迟。
- 各类内存有明确的用途：寄存器存放线程私有数据，共享内存用于块内通信，常量/纹理缓存优化只读访问，全局内存作为主存储。
- 与 CPU 相比，GPU 在近端内存容量上占优，CPU 在远端缓存和单线程延迟上占优。
